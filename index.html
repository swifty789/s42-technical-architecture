Signal OS — Full Memory RAG Architecture

This is the top-level system diagram. It shows all three core services, their internal components, the shared infrastructure layer, and every external API dependency.

The architecture follows a service-oriented pattern where the Memory Service acts as the primary orchestrator — it receives user messages, assembles context from multiple sources, and delegates document retrieval to the RAG Engine. The Doc-Ingestion Service operates semi-independently, processing uploaded documents through an 11-node LangGraph pipeline.

All three services communicate via HTTP APIs over a shared Docker network. Infrastructure services (MongoDB, PostgreSQL, Redis, MinIO) are containerised alongside the application services.

Memory Service — Context Assembly

The Memory Service is the brain of Signal OS. Its core component is the Data Controller (app/services/data_controller.py), which assembles a personalised context object for every AI interaction.

What makes this architecture distinctive is the Service Manifest system. Each AI service declares a YAML manifest specifying exactly what context it needs. This selective fetching keeps context lean and relevant, avoiding the "throw everything at the LLM" antipattern.

The learning loop is equally important: after every response, LangMem extracts patterns and writes them back to Mem0, so the system gets smarter over time.

Doc-Ingestion — 11-Node LangGraph Pipeline

The Doc-Ingestion Service transforms raw documents into high-quality, RAG-ready chunks via an 11-node LangGraph pipeline (app/graph/graph.py).

The pipeline's most notable design feature is conditional routing at the size_check node. Documents with images go through a vision node (Gemini 3 Flash), documents with tables go through table extraction, and text-only documents skip straight to AI metadata extraction.

Every major stage saves a checkpoint to PostgreSQL, allowing resume from the last successful stage on failure. The pipeline ends with a manual approval gate. Batch processing uses a Celery worker backed by Redis. Files are stored in MinIO.

RAG Engine — 5-Node Query Pipeline

The query pipeline is a 5-node LangGraph graph implementing: embed → search → rerank → compress → generate.

Embed uses Voyage AI's voyage-multimodal-3.5 (1024-dim). Search performs hybrid retrieval against Pinecone (dense + sparse BM25). Rerank uses Jina AI. Compress reduces context to fit token limits. Generate routes to the appropriate LLM.

The pipeline returns a confidence score plus full source metadata for attribution. LangSmith traces every run.

Full Data Flow — Upload to Answer

This sequence diagram shows the complete end-to-end journey: document ingestion and query answering.

Phase 1 (Ingestion): Upload → MinIO → 11-node pipeline → PostgreSQL checkpoints → human approval → RAG Engine → Voyage AI embeddings → Pinecone.

Phase 2 (Query): Question → Memory Service → MongoDB context → service manifest → RAG Engine → embed → hybrid search → rerank → compress → LLM → response with sources → LangMem learning extraction.

Infrastructure & Deployment

The entire platform runs as a Docker Compose stack: 6 application services and 4 infrastructure services. External AI services are called over HTTPS. Production targets Fly.io.

Key dependency chains: Doc-Ingestion depends on PostgreSQL, Redis, and MinIO. Memory depends on MongoDB. MinIO has an init container that creates required buckets on first run.

LLM Model Routing

The LLM Router (packages/rag-engine/backend/app/services/llm_router.py) routes requests based on task type with automatic fallback and exponential backoff.

Task-specific model selection: Gemini 3 Flash for extraction, vision, and generation (cost/speed). Claude Sonnet 4.5 for validation (accuracy). Each has defined fallbacks.

Provider-specific parameters: Anthropic uses standard temperature. OpenAI uses verbosity + reasoning_effort. Google must use temperature=1.0 and uses thinking_level instead. The router handles these mappings transparently.
